# ADR-0008 — Таксономия исключений библиотеки

- **Статус:** Proposed
- **Драйверы:** D6, NFR-9 (стабильный API); находка ревью G4.
- **Воркстрим:** M0 (база) + M1 (доменные).

## Контекст

Сейчас обработка ошибок грубая: `except Exception` + `print` + общий `RuntimeError`
(as-is D4). Именованное исключение ровно одно — `MissingDependencyError`
(ADR-0001). Для публичной OSS-библиотеки иерархия исключений — **часть публичного
API** (под SemVer, NFR-9): пользователь ловит конкретные типы; «дорастить»
иерархию позже без breaking нельзя.

## Решение

Единая иерархия с базой `AutoMLError(Exception)`. Никаких голых
`except Exception`/`RuntimeError` в библиотечном коде; на границах — конкретные
типы.

```
AutoMLError                      # база, M0
├── ConfigError                  # невалидная конфигурация (Pydantic-обёртка), M0
├── MissingDependencyError       # отсутствует optional extra (ADR-0001), M0
├── SchemaValidationError        # вход не соответствует FeatureSchema/Task, M1
│   └── (длины X/y, неизвестные колонки, y вне Task.kind, all-NaN, дрейф dtype)
├── BudgetExhaustedError         # исчерпан бюджет (время/память), M1/M5
├── FitFailedError               # падение обучения конкретной комбинации, M3/M5
└── NotFittedError               # вызов predict до fit, M1
```

- **M0** вводит базу `AutoMLError`, `ConfigError`, и переносит
  `MissingDependencyError` под неё.
- **M1** добавляет доменные: `SchemaValidationError`, `NotFittedError`,
  `BudgetExhaustedError` (заготовка под G7/память).
- `FitFailedError` (M3/M5) заменяет нынешнее «накопить `failed_combos` и бросить
  общий `RuntimeError`»: падение одной комбинации изолируется, агрегируется и
  репортится явным типом (resilience, P16).
- Каждое исключение несёт структурный контекст (поле/комбинация/лимит), не только
  строку.

## Последствия

- (+) Пользователь ловит конкретные типы; иерархия зафиксирована до 1.0 (NFR-9).
- (+) Снимает D4 (грубый `except Exception`/`print`).
- (−) Дисциплина: каждый слой бросает свой тип; маппинг внешних исключений
  (catboost/optuna) в доменные — в адаптерах.
- Связь: `SchemaValidationError` — носитель контракта валидации входа (G5);
  `BudgetExhaustedError` — для memory/time-бюджета (G7, наполнение M5).

## Поправки / связанные ADR

- **ADR-0014 (воркстрим F1)** добавляет новый **лист** `InputError(AutoMLError)` —
  для I/O-режимов отказа loader (нет файла, нечитаемо, кодировка). Расхождение схем в
  папке остаётся `SchemaValidationError`. Добавление листа **аддитивно**: существующие
  `except`-ы по конкретным типам и по `AutoMLError` не ломаются (это не «расширение
  базы», против которого направлено решение выше), совместимо с SemVer-намерением.
- **ADR-0016 (воркстрим C4)** добавляет лист `UnsupportedSchemeError(ConfigError)` —
  «схема CV/параметр валидны, но ещё не реализованы (M3/M4)». Подтип `ConfigError`:
  ловится и как `ConfigError`, и как `AutoMLError`; отделяет «не реализовано» от
  «невалидно». Аддитивно (лист, не расширение базы).
- **ADR-0019 (воркстрим M3)** добавляет лист `PluginConflictError(ConfigError)` —
  дубль имени компонента в группе реестра (fail-fast при сборке). Аддитивно (лист).
- **ADR-0022 (воркстрим M3)** наполняет `FitFailedError` и **уточняет семантику**:
  падение одного кандидата/фолда **не** бросает `FitFailedError` (логируется и сворачивается
  в `SliceResult.failed`); `FitFailedError` — **терминальный** случай «**все** кандидаты не
  обучились». (Это сужает строку дерева «падение обучения конкретной комбинации»: per-candidate
  изоляция вместо немедленного броска.)

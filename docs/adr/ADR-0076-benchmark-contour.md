# ADR-0076 — Бенчмарк-контур: верификация честности, сравнимая между релизами

- **Статус:** Accepted
- **Драйверы:** DM-D4 (02-drivers.md); FR-19 (carry-in G-B1), рамка north-star §4.0

## Контекст

Главный claim проекта — «честно лучшая модель» — неверифицируем (G-B1: «одна строка
без корпуса»). Рамка north-star §4.0 явно запрещает гонку за raw-метрикой: бенчмарк
должен мерить **честность** (оценка, которую библиотека сообщает при отборе, не
оптимистична относительно нетронутого holdout), и быть сравнимым между релизами.
TBC-пороги north-star NFR (например бюджетный допуск NFR-7) финализируются по этим
данным.

## Варианты

1. **Локальный оффлайн-контур `benchmarks/` (не шипится): детерминированный корпус →
   протокол selection-vs-holdout → committed baseline + сравнение в допуске.** ✅
2. AMLB/OpenML-интеграция — сетевые загрузки и часы CI; ценно как Day-2-расширение,
   не как гейт. Отложено (operational).
3. Перф-бенчмарк (скорость/память) — другой вопрос; не верифицирует claim. Вне объёма.

## Решение

### §1. Место и состав

`benchmarks/` в корне (вне `src/` — в wheel не попадает): `corpus.py` (декларативный
список датасетов), `run.py` (раннер + `--check`/`--update-baseline`), `baseline.json`
(committed эталон), `README.md` (протокол). Зависимости — только `[dev,boosting]`.

### §2. Корпус (NFR-DLV-3: оффлайн, детерминирован)

Только встроенные sklearn-датасеты + синтетика с фиксированными seed (никакой сети):
binary (breast_cancer; make_classification: базовый / редкий класс / шумные фичи),
multiclass (wine; make_classification 4 класса), regression (diabetes;
make_regression + skewed-таргет). ~8 датасетов, каждый ≤ нескольких тысяч строк —
полный прогон минуты, не часы. Датасет = запись «имя → loader-вызов + task + seed».

### §3. Протокол честности

Для каждого датасета: `AutoML(task, random_state=SEED, cv=CVConfig(outer_holdout=0.2),
models=None)` — `fit` в full-режиме, БЕЗ timeout-зависимых опций (детерминизм,
R-DLV-3). Записывается: winner, `selection_score` (сырой скор победителя из
leaderboard), `holdout_score`, band-факты, состав моделей, версии.

**Формула (ориентация применяется раннером, скоры в отчёте сырые):**
`optimism_d = (selection_score − holdout_score) · (+1, если greater_is_better, иначе −1)`
(`greater_is_better` — из метрики датасета, объявленной в corpus.py). Агрегаты —
**по семействам метрик** (classification-AUC-шкала и regression-RMSE-шкала не
смешиваются в одно среднее), репортуются mean/max optimism per family.

**Вердикт (гейт):** отсутствие РЕГРЕССА — per-dataset сравнение
`optimism_d`/`holdout_score` с committed baseline в per-dataset допусках из
baseline.json. Абсолютный порог оптимизма НЕ является гейтом v1 (стартовый прогон не
может сам себя сертифицировать — циркулярность): mean/max optimism — репортуемая
диагностика, абсолютная граница фиксируется позже, по накопленным данным (TBC-механизм
north-star NFR).

### §4. Сравнимость между релизами и CI

`run.py` пишет `results.json` без таймингов (NFR-DLV-3: два прогона в одном
окружении → побайтно одинаково; same-env тест двойного прогона на одном маленьком
датасете живёт в unit-suite, помечен `slow`). **Провенанс baseline:**
`baseline.json` генерируется и обновляется ТОЛЬКО CI-джобой (pinned ubuntu + python,
`uv sync --frozen`) — локальная Windows/BLAS-вариация не попадает в эталон; политика
и правило выбора per-dataset допусков — в benchmarks/README. `--check` сравнивает с
baseline; `--update-baseline` — осознанное действие с записью в CHANGELOG.
CI: **отдельный `benchmark.yml`** (`workflow_dispatch`; НЕ в ci.yml — иначе джоба
запускалась бы push/PR-триггерами или потребовала if-гардов) + ручной пункт
релизного чек-листа (ADR-0077 §3).

## Последствия

- (+) Claim верифицируем и регрессионно защищён; данные для финализации NFR-3 TBC.
- (+) Ноль новой инфраструктуры/сети; корпус расширяем записью в `corpus.py`.
- (−) Малый встроенный корпус ≠ репрезентативность AMLB; это контур честности, не
  лидерборд против конкурентов (так и задумано, §4.0). OpenML-расширение — Day-2.

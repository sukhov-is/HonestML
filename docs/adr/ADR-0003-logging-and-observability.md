# ADR-0003 — Логирование и наблюдаемость

- **Статус:** Proposed
- **Драйверы:** D6; NFR-10, D2 (`print` по всему коду).
- **Воркстрим:** M0.

## Контекст

Сейчас прогресс и диагностика печатаются через `print(...)` по всем модулям
(`data`, `selection`, `pipeline`, `tuning`, `tracking`). Потребитель библиотеки
не может управлять уровнем/направлением вывода, нет структуры, вывод засоряет
stdout в ноутбуках/сервисах.

## Рассмотренные варианты

1. **Оставить `print`** — нет.
2. **stdlib `logging`** с библиотечным неймспейсом и `NullHandler` — канонический
   способ для библиотек: потребитель сам решает, как обрабатывать.
3. **structlog/loguru** — удобнее, но навязывает зависимость и стиль потребителю;
   для ядра OSS-библиотеки избыточно.

## Решение

Вариант **2**:
- Корневой логгер библиотеки `logging.getLogger("honestml")`, по умолчанию с
  `NullHandler` (никакого вывода, пока потребитель не настроит хендлеры).
- Уровни: `DEBUG` (детали этапов), `INFO` (прогресс комбинаций/этапов),
  `WARNING`/`ERROR` (деградации/сбои). Никаких `print` в библиотечном коде
  (проверяется lint-правилом/тестом отсутствия `print`).
- Сообщения — структурированные `key=value` (combo, stage, metric, elapsed) для
  машинного парсинга; помощник форматирования в утилитах.
- Прогресс как **наблюдаемый артефакт**: помимо логов — опциональный
  callback-хук (`on_event`) у use-case прогона, чтобы UI/ноутбук/трекер
  подписывались без парсинга строк. Тяжёлый прогресс-бар (tqdm) — опционален.

## Последствия

- (+) Потребитель управляет уровнем/направлением; чистый stdout по умолчанию.
- (+) Структурные поля упрощают трекинг и отладку.
- (−) Нужно заменить все `print` и провести логгер/контекст через слои (вместе с
  миграцией ADR-0002).
- Callback-контракт `on_event` проектируется детально в M5 (оркестрация).

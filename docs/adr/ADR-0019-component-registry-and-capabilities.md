# ADR-0019 — Реестр компонентов на entry-points + capability-декларации

- **Статус:** Accepted (реализован 2026-06-08, M3a)
- **Дата:** 2026-06-07
- **Драйверы:** DM3-1 (расширяемость через реестр), DM3-2 (ленивость/упаковка), DM3-6 (детерминизм
  дискавери); FR-1, FR-2; NFR-1, NFR-2, NFR-3, NFR-4, NFR-5. north-star: D1/D6.
- **Воркстрим:** M3, дельта к M2. Расширяет ADR-0006 (`Capabilities`), ADR-0009 (composition root).

## Контекст

`build.py:71-79` хардкодит словарь фабрик `{"dummy","linear"}`; метрики — `_REGISTRY` dict
(`metrics.py:80`); сплиттеры — ручной `_resolve_splitter`. Нет дискавери и нет пути для стороннего
компонента без правки ядра (подрыв D1). `ModelSpec`/`Capabilities` объявлены (ADR-0006), но не
используются; фильтр способностей `_filter_by_capability:152-163` **материализует** `factory()` ради
чтения `.capabilities` и `isinstance(probe, ProbabilisticEstimator)` — для тяжёлых бустингов это
импорт `catboost` ради фильтра (ломает ленивость, RM3-2). Перечисление entry-points не детерминировано
и не дедуплицировано (RM3-1).

## Рассмотренные варианты

1. **Оставить хардкод-словари, добавить бустинги inline** — не масштабируется, нет сторонних плагинов,
   тяжёлый импорт на старте; нарушает D1/NFR-2.
2. **Чистые entry-points для всего, включая встроенные** — единообразно, но встроенные начинают зависеть
   от корректности метаданных пакета даже в dev-режиме; дискавери на каждом старте читает metadata для
   своих же модулей. Хрупко для тестов/редактируемой установки.
3. **Гибридный реестр-абстракция** — единый `ComponentRegistry`, сливающий (a) встроенные дескрипторы,
   объявленные внутри пакета, и (b) сторонние через `entry_points(group=...)`; оба выражены лёгким
   `Descriptor` (имя + статич. `Capabilities` + ленивый `build()`); фильтр по статич. тегам; fail-fast на
   дублях; детерминированная сортировка; материализация (`build()`) только выживших. **Выбран.**

## Решение

**Вариант 3.**

### 1. `ComponentDescriptor` (тип — в `composition/registry.py`; экземпляры — в `adapters/`)
Каноническое имя — **`ComponentDescriptor`** (везде единообразно). Лёгкий объект (dataclass, не pydantic —
несёт `Callable`):
```
ComponentDescriptor:
    name: str
    spec: ModelSpec            # имя + Capabilities (+ search_space=  {} для M3, заполнит M7)
    build: Callable[..., T]    # ленивая фабрика; тяжёлый import ВНУТРИ build, не на уровне модуля
    api_version: int = 1       # версия плагин-контракта (фикс R2-A1); неизвестная → skip+WARNING (§6)
```
**Поправка реализации (2026-06-08, M3b).** Дескриптор получает аддитивное поле
`requires: tuple[str, ...] = ()` — рантайм-модуль(и), нужные компоненту (пусто = всегда доступен).
`ComponentRegistry.is_available(name)` проверяет их через `importlib.util.find_spec` (**без импорта**
тяжёлой либы). Это нужно дефолтному отбору в `build` (см. ADR-0020 §5): при `models=None` бустинг
авто-включается только если его extra установлен; явный выбор отсутствующего → `MissingDependencyError`.
Поле и проверка аддитивны (дефолт `()` → поведение без extras не меняется), детерминизм/дедуп сохранены.
Контракт зафиксирован в `docs/plugin-contract.md`.

**Дом (фикс R1-F4):** *тип* `ComponentDescriptor` определён в `composition/registry.py` (рядом с реестром);
*экземпляры* встроенных компонентов объявляются в `adapters/` (модели/метрики/сплиттеры). Слои соблюдены
(composition→adapters→core). Для моделей `T = Estimator`.

**Граница ленивости (фикс R1-F1).** Дескриптор **не импортирует** тяжёлую зависимость на уровне модуля —
`import catboost` живёт внутри `build()`. Для **встроенных** это гарантируем мы (тест `sys.modules`,
NFR-2). Для **сторонних** ленивость держится на контракте: «модуль с `ComponentDescriptor` обязан быть
import-light; тяжёлый импорт — только в `build()`». Это **граница ответственности плагина**, не наша
гарантия — фиксируется в docs/плагин-контракте; сторонний дескриптор, импортирующий тяжёлый пакет на
загрузке, нарушает контракт (поведение плагина). Принято как явный риск (RM3-2): защитного sandbox-механизма
не вводим (избыточно для M3); наша часть (встроенные + тест-плагин) проверена.

### 2. `Capabilities` += `probabilistic` (аддитивно, ADR-0006)
В `core/ports/model_spec.py` добавляется поле `probabilistic: bool = False` (умеет ли `predict_proba`).
Дефолт `False` → forward/backward-совместимо (NFR-3). Фильтр читает `spec.capabilities.probabilistic`
**без** материализации (закрывает RM3-2). `Capabilities.model_config` остаётся `frozen, extra="forbid"`;
поле аддитивно (не ломает существующие `_BINARY_CAPS`).

### 3. `ComponentRegistry` (новый модуль `composition/registry.py`)
Единый дискавери, обобщённый по группе:
- **Группы:** `honestml.models`, `honestml.metrics`, `honestml.splitters` (константы).
- **Встроенные:** статический список дескрипторов на пакет (встроенные модели/метрики/сплиттеры
  регистрируются через тот же `Descriptor`-тип; для M3 источник встроенных — внутренний модуль, не
  обязательно реальные entry-points, чтобы dev/editable-установка работала без metadata).
- **Сторонние:** `importlib.metadata.entry_points(group=...)`; **`.load()` не вызывается** на дискавери —
  читаются только метаданные (`name`). Для capability-фильтра нужен дескриптор → загрузка дескриптора
  (`ep.load()`) лёгкая (модуль-дескриптор не тянет тяжёлую зависимость); тяжёлый импорт отложен до `build()`.
- **Слияние и конфликты (порядок шагов фиксирован, фикс R1-4a):** (1) собрать все дескрипторы (встроенные +
  сторонние), (2) **dedup/конфликт-детект по имени ДО сортировки** — дубль (встроенный↔встроенный,
  сторонний↔сторонний, сторонний поверх встроенного) → **`PluginConflictError`** (новый лист, см. §5),
  fail-fast, не «последний выигрывает» (RM3-1); (3) сортировка выживших. Политику «сторонний переопределяет
  встроенный» отвергаем (тихие сюрпризы).
- **Детерминизм (фикс R1-4a, R2-A2):** поскольку дубли имён отсеяны на шаге (2), **`name` уникален** —
  первичная сортировка **по `name`** уже даёт тотальный детерминированный порядок, не зависящий от обхода
  `sys.path` (RM3-6/NFR-4). `dist_name` (синтетический `"<builtin>"` у встроенных; реальный или `None` у
  сторонних в editable-без-metadata) используется лишь как стабильный вторичный ключ и **не влияет** на
  детерминизм (его `None` безопасен, т.к. `name` уже разрешает порядок).
- **Материализация:** `registry.build(name, **kwargs)` зовёт `descriptor.build(...)`; `ImportError`/
  `ModuleNotFoundError` внутри → **`MissingDependencyError`** с именем недостающего пакета (NFR-5).

### 4. `build_default_components` переходит на реестр
`build.py` больше не хардкодит словарь моделей: берёт дескрипторы из `ComponentRegistry(group=models)`,
фильтрует по `Task.kind` и `metric.needs` **на статич. `spec.capabilities`** (proba-метрика требует
`capabilities.probabilistic`), выбирает подмножество по `models: tuple[str,...]` (как раньше), и отдаёт
`estimators: dict[name, EstimatorFactory]`, где фабрика = `lambda: registry.build(name, task=task,
random_state=...)`. Порядок отказов сохраняется (ADR-0016 §2): фильтр способностей до CV-резолва;
пусто после фильтра → `ConfigError`. Неизвестное имя в `models` → `ConfigError` со списком доступных.
`_filter_by_capability` (материализующий) удаляется/заменяется на фильтр по дескрипторам.

### 5. Новые исключения
`PluginConflictError(ConfigError)` (дубль имени) — аддитивно к ADR-0008, ловится и как `ConfigError`.
`MissingDependencyError` уже существует (используется при отсутствии extra на `build()`).

### 6. Версионирование контракта
Реестр несёт `REGISTRY_API_VERSION = 1`. Дескриптор объявляет свою `api_version` (поле, §1, дефолт 1).
Реестр сверяет `descriptor.api_version` с поддерживаемым диапазоном: неизвестная (будущая) версия →
**skip + WARNING с именем плагина** (не краш реестра; фикс R2-A1 — теперь механизм реализуем, версия
читается из поля дескриптора). Новые capability-поля — только аддитивно с дефолтами (NFR-3).

### 7. Read-only интроспекция реестра (фикс R2-C4)
`registry.descriptors(group) -> list[ComponentDescriptor]` и фасадный хелпер `AutoML.available_models(
task=None)` -> имена+capabilities — **без материализации** (читают дескрипторы). Закрывает «как пользователь
узнает доступные/сторонние компоненты», особенно бустинги под extras. Дёшево (реестр уже всё имеет);
питает docs/quickstart (NFR-12).

### 8. Плагин-контракт как deliverable (фикс R2-C5)
Граница ответственности плагина (что реализовать: `ModelSpec`+`Capabilities`+`build`; группы entry-points;
правило import-light; `api_version`; депрекация) фиксируется в **`docs/plugin-contract.md`** (пишется в
implementation). Без письменного контракта NFR-3 (стабильность) непроверяема. Трассируется под NFR-3/NFR-12.

## Последствия

- **Положительные:** D1 реализован (сторонний компонент без правки ядра); ленивость сохранена (фильтр без
  материализации, тяжёлый импорт только у выбранного); детерминизм и fail-fast на конфликтах; встроенные и
  сторонние — единый путь; `ModelSpec`/`Capabilities` перестают быть мёртвыми.
- **Отрицательные / компромиссы:** появляется `composition/registry.py` + зависимость от `importlib.metadata`
  (infra, не в `core`); встроенные дескрипторы дублируют то, что раньше было inline-словарём (плата за
  единый механизм); дубль-имя теперь жёстко падает (намеренно). Сторонний дескриптор-модуль обязан НЕ
  импортировать тяжёлую зависимость на верхнем уровне — это контракт, описан в docs (NFR-2/-3).
- **Влияние на слои:** реестр — в `composition` (не в `application`: `import-linter`
  `usecases-independent-of-adapters` сохраняется); `core` чист от entry-points; `Capabilities.probabilistic`
  — домен (аддитивно). import-linter 3/3.

## Проверки

Тест-плагин через monkeypatch `entry_points` (model/metric/splitter) → участвует в прогоне без правки
`core`/`composition`; дискавери без установленного extra не импортирует тяжёлый пакет (проверка `sys.modules`)
и не падает; capability-фильтр отсекает по `tasks`/`probabilistic` **без** материализации; дубль имени →
`PluginConflictError`; перемешанный источник entry-points → детерминированный порядок кандидатов; выбор
отсутствующей модели → `MissingDependencyError`; неизвестное имя в `models` → `ConfigError`;
`Capabilities` со старым набором полей (без `probabilistic`) валиден (дефолт); `import-linter` 3/3 в CI.

# ADR-0087: Capability-driven роутинг представления категорий

- **Статус:** Proposed
- **Дата:** 2026-06-22
- **Драйверы:** D-1, D-7 / FR-1, FR-2

## Контекст
Сейчас `design_matrix` строит единый `numeric ⊕ codes(float64)` и подаёт его
**всем** кандидатам (`slice.py:180-207, :548, :758`); категории — ordinal-коды как
числа. CatBoost/LightGBM умеют категории нативно и обычно точнее на
средне-/высококардинальных признаках (`correctness-guide.md:96-99`). Поле
`Capabilities.handles_cat` существует, но захардкожено `False` и не читается ни
отбором, ни обучением (`model_spec.py:31`, `registry.py`). Нужно включить роутинг:
native-capable модели — нативно, остальные — как сейчас, **не ломая** общий путь и
честность.

## Рассмотренные варианты
1. **Per-model препроцessing-пайплайны** (каждый кандидат строит свой вход) —
   гибко, но ломает «один `design_matrix` для всех» (ADR-0045), дублирует
   материализацию, усложняет честность и инференс. Избыточно для двух бэкендов.
2. **Глобальный флаг «native cat везде»** — нельзя: linear/baseline/xgboost
   нативно категории не умеют; сломает их и сравнимость.
3. **(выбрано) Capability-роутинг поверх единого `design_matrix`:** матрица
   остаётся общей; native-capable моделям **дополнительно** передаётся список
   категориальных индексов; флаг способности — `handles_cat` (статически) +
   рантайм-маркер `SupportsNativeCategorical` (ADR-0088). Остальные модели не
   получают индексов и работают как прежде.

## Решение
Активировать `handles_cat` **per-backend** и ввести роутинг по способности:
- Расщепить общий `_BOOST_CAPS`: `handles_cat=True` для `catboost` и `lightgbm`,
  `False` для `xgboost`; `linear`/`baseline` остаются `False` (D-7, R-8).
- `run_slice` для кандидата, реализующего `SupportsNativeCategorical` (ADR-0088),
  передаёт категориальные индексы; иначе — ничего не меняет. Решение принимается
  по **способности модели**, не по строковому имени бэкенда в use-case (use-case
  остаётся адаптер-слепым, контракт ADR-0009).
- **Граница** slice↔adapter (`design_matrix`, float64-numpy, ADR-0005) и путь
  кодов **не меняются**. Нативная обработка — это (1) дополнительные индексы
  категориальных колонок, передаваемые адаптеру, и (2) **материализация входа
  внутри адаптера** (ADR-0089: CatBoost — int-каст cat-блока; LightGBM —
  `categorical_feature` поверх той же float64-матрицы). На границе матрица
  остаётся прежней; иной вид получает только сам нативный estimator внутри обёртки.

Точность представления категорий (нативно vs коды) становится, таким образом,
**свойством кандидата**, как `needs_scaling` у linear, а не глобальным режимом.

## Последствия
- **Положительные:** минимальная инвазивность (общая матрица сохранена); единый
  механизм для будущих native-capable моделей; `handles_cat` из мёртвой
  декларации становится действующей способностью; нулевая регрессия не-нативных
  путей (NFR-3).
- **Отрицательные / компромиссы:** появляется второй «канал» данных к адаптеру
  (индексы рядом с матрицей) — оформляется явным портом (ADR-0088), чтобы не
  превратиться в скрытую связанность; `_BOOST_CAPS` перестаёт быть единой
  константой для трёх бэкендов.
- **Влияние на слои/границы:** способность — в `core/ports`; роутинг — в
  application (`run_slice`); расщепление caps — в composition (`registry.py`).
  Зависимости только внутрь; новый контракт import-linter не нужен (см.
  `05-design/dependency-graph.md`).

## Проверки
- Тест: на категориальном датасете catboost/lightgbm получают непустые
  категориальные индексы, остальные — нет (FR-1).
- Тест: `available_models()` → `handles_cat=True` только у catboost/lightgbm;
  xgboost/linear/baseline = `False` (FR-2).
- `uv run pytest` без изменения ожиданий не-нативных моделей (NFR-3).

# ADR-0095: Наблюдаемость решений роутинга (per-column verdict)

- **Статус:** Accepted (реализовано 2026-06-23)
- **Дата:** 2026-06-23
- **Драйверы:** D-4 (каждое честное решение наблюдаемо), FR-5, NFR-4

## Контекст
Гейт меняет, какие колонки идут нативно. По принципу HonestML каждое такое решение должно быть
**видимым** (как no-selection/ensemble/calibration гейты), иначе «тихая» демоция колонки выглядит
как «всё нативно», когда это не так. Силы: переиспользовать существующий контур отчёта/манифеста
без bump версии (NFR-4); словарь причин уже есть.

## Рассмотренные варианты
1. **Per-column verdict в run-report + отражение в manifest, по образцу `no_selection_gate`.**
   Аддитивный блок в `run_report_` со списком демотированных колонок и причиной (словарь
   `high_cardinality`, согласованный с `TypingDecision`: `low_cardinality_int`/
   `high_cardinality_string`/`numeric_id_like`); `native_categorical.n_cat` в manifest = число
   **нативных после гейта**; info/warning-лог на демоции. Плюсы: однороден с существующими гейтами;
   аддитивно (без bump `RUN_MANIFEST_VERSION`/`artifact_version`); переиспользует словарь. Минусы:
   ещё один блок отчёта (приемлемо — как все прочие).
2. **Только лог, без записи в отчёт.** Минусы: не воспроизводимо/не машиночитаемо; расходится с
   тем, как фиксируются другие гейты (D-4). Отвергнут.
3. **Bump версии манифеста под новые ключи.** Минусы: не нужен — все прочие блоки отчёта аддитивны
   без bump; сломал бы загрузку старых артефактов зря (NFR-4). Отвергнут.

## Решение
**Вариант 1.** Решения роутинга фиксируются как **аддитивный per-column verdict**:
- **Носитель (ревью R1-#3):** гейт возвращает не только имена, но и per-column verdict; он кладётся
  на **новое поле `SliceResult.native_routing`** (структурированный результат, заполняемый в
  `run_slice` из вывода гейта) — ровно как `selection_gate` едет на `SliceResult.feature_selection`.
  Без этого носителя `build_run_report` нечего surface'ить (он читает из `SliceResult`).
- в `run_report_` — `build_run_report` отражает `native_routing` как новый аддитивный раздел (какие
  категориальные ушли нативно, какие демотированы и почему), без bump `RUN_MANIFEST_VERSION`;
- в артефакте — существующий `native_categorical {backend, n_cat}` отражает **отгейченный** счётчик
  (`n_cat` = число нативно-маршрутизированных после гейта; следствие фикса проводки `refit_best`
  из ADR-0092 — иначе manifest пишет до-гейтный счёт, ревью R1-#2), аддитивно, без bump
  `artifact_version`;
- на демоции ≥1 колонки — **`logger.warning`** (один уровень, как drop-лог `no_selection_gate` —
  ревью R2-#4).

**Словарь причин (ревью R1-#4):** `TypingDecision` живёт в `adapters/reader.py` (адаптер), и
application/composition его импортить **не могут** (`usecases-independent-of-adapters`/`layers`).
Поэтому verdict использует **собственный** reason-токен, определённый **там же, где гейт** (core,
рядом с `native_routable`) — напр. `Literal["native", "high_cardinality"]`. Это **не** переиспользование
adapter-локального `NamedTuple` и не дублирование магических строк через границу слоя; язык остаётся
согласованным (термин `high_cardinality` совпадает по смыслу со словарём типизации), но источник — один,
в core.

## Последствия
- **Положительные:** прозрачность (видно, что и почему ушло в коды); машиночитаемо и воспроизводимо;
  однородно с прочими гейтами; нулевой bump версий (NFR-4).
- **Отрицательные / компромиссы:** +1 блок отчёта и +лог — минимальная поверхность.
- **Влияние на слои/границы:** запись verdict — в application (отчёт) и composition (manifest); core
  лишь возвращает решение гейта. Слой не нарушается.

## Проверки
`test_native_serialization.py`/`test_run_slice.py`: при демоции high-card колонки `run_report_`
несёт блок с её именем и причиной; `native_categorical.n_cat` = число нативных после гейта;
лог-сообщение фиксируется (caplog); старый артефакт без блока грузится (NFR-4).

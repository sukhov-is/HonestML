# ADR-0022 — Изоляция падения кандидата/фолда (FitFailedError)

- **Статус:** Accepted (реализован 2026-06-08, M3a)
- **Дата:** 2026-06-07
- **Драйверы:** DM3-5 (резилентность прогона); NFR-6. north-star: D7; NFR-15 (carry-in D4).
- **Воркстрим:** M3, дельта к ADR-0010 (`run_slice`). Расширяет ADR-0008 (исключения).

## Контекст

`run_slice:157-173` собирает кандидатов list-comprehension `[_run_candidate(...) for ...]` — **без
изоляции**: исключение в `_run_candidate` (`est.fit` стр. 228, `predict`/`predict_proba`) валит **весь
прогон**, остальные модели не запускаются, пользователь видит стек вместо «модель X упала, пропущена»
(D4, live-дефект; критично до зоопарка моделей, где падения вероятнее). Внутри фолда уже есть мягкий
skip только для «single-class test», но не для исключений fit/predict.

## Рассмотренные варианты

1. **try/except только на уровне фолда** — частичный OOF при падении части фолдов, но не изолирует
   полностью неработающую модель (например, несовместимый вход) — она всё равно бросит на каждом фолде.
2. **try/except только на уровне кандидата** — модель целиком failed или ok; проще, но теряет частичную
   устойчивость к редкому падению одного фолда.
3. **Двухуровневая изоляция: фолд + кандидат** — исключение фолда → skip фолда + WARNING; если у кандидата
   ни один фолд не дал OOF (или фабрика/скоринг упали) → кандидат **failed** (исключён из leaderboard);
   все кандидаты failed → **`FitFailedError`**. **Выбран** (резилентность + честный отказ).

### 1. Изоляция — **per-candidate**, не частичное скорение фолдов (фикс R1-B1)
Кандидаты сравниваются `rank`/`select_best` по **абсолютному скаляру**, посчитанному на OOF-маске. Если у
кандидата A упали 2 из 5 фолдов, его скаляр считается на ≈60% строк, у кандидата B (все фолды) — на 100%;
скаляры **несопоставимы** (разные тестовые подмножества) → битый кандидат может «выиграть». Поэтому:
**исключение `factory()`/`fit`/`predict`/`predict_proba` в любом фолде → весь кандидат помечается failed**
(не частичное OOF-скорение). WARNING (`model=…, fold=…, error=…`). Так покрытие у всех **успешных**
кандидатов одинаково (его задаёт только общий для всех `single-class test`-skip, зависящий от `y[test_idx]`,
а не от модели) → `rank` честен. Частичный OOF (для значимости) — задача M4, не primary-скоринга M3.

### 2. Узкий try/except — только вокруг внешней модели (фикс R1-3a, R2-уточнение)
try/except оборачивает **строго** вызовы внешнего кода: `factory()`, `est.fit`, `est.predict`,
`est.predict_proba`. **Наш numpy-код** (`align_proba`, `project_for_metric`, индексация по `classes_`,
`isinstance`-инвариант, `metric.score`) **вне** except — его баги падают явно, а не сворачиваются в ложный
«failed candidate» (инвариант проекта: ловим на границе, не глушим доменную логику).
**Паттерн внутри фолд-витка (явно, фикс R2):** `try:` захватывает только сырые выходы модели в локальные
(`raw_proba = est.predict_proba(x_test)` / `raw_pred = est.predict(x_test)`); `except` → пометить кандидат
failed и прервать его фолды; **после `try`** (в том же витке) — `align_proba(raw_proba, est.classes_,
classes)` и накопление в OOF (наш код вне изоляции). `project_for_metric`/`metric.score` — после фолд-цикла
(вне и цикла, и except). Так «внешнее» и «наше» физически разнесены, изоляция не расплывается.
Пост-материализационный инвариант (ADR-0020 §3.2): `isinstance(est, ProbabilisticEstimator)` для proba-метрики
проверяется **вне** except (наш код); нарушение → явный skip+WARNING/ошибка, не «тихий failed».

### 3. Все упали → явная ошибка
Если после перебора **нет ни одного успешного** кандидата → `FitFailedError` со сводкой (имена + краткие
причины). Это закрывает RM3-5 (изоляция не маскирует «всё сломано»).

### 4. Результат несёт failed-список
`SliceResult` получает аддитивное поле `failed: list[FailedCandidate]` (`id: str`, `reason: str`) — для
наблюдаемости/манифеста/отчёта (M5/M9). `leaderboard` — только успешные. Совместимо: поле аддитивно,
дефолт `[]`.

### 5. Новое исключение
`FitFailedError(AutoMLError)` — новый лист (ADR-0008): «все кандидаты не обучились». Per-candidate ошибки
**не** поднимаются как `FitFailedError` (они логируются и сворачиваются в failed-список); `FitFailedError`
— только терминальный «все упали».

## Последствия

- **Положительные:** один битый эстиматор не рушит прогон (NFR-6/NFR-15); результат — по выжившим;
  сравнение кандидатов **честное** (одинаковое OOF-покрытие у успешных, фикс R1-B1); «всё упало» — явная
  диагностируемая ошибка, не тихий пустой leaderboard; failed-список → наблюдаемость; баги нашего кода
  падают явно, не маскируются (фикс R1-3a).
- **Отрицательные / компромиссы:** падение даже одного фолда у кандидата → весь кандидат failed (теряем
  устойчивость к одному «флапающему» фолду ради честности сравнения — осознанный выбор; частичный OOF —
  M4); `except Exception` **узкий** (только вокруг внешней модели), что намеренно (модель — внешний код);
  `SliceResult` меняет форму (аддитивно).
- **Влияние на слои:** изменения — `application/slice.py` (use-case), `core/exceptions.py`
  (`FitFailedError`), `core` модель `FailedCandidate`/поле `SliceResult`. import-linter не нарушен.

## Проверки

Эстиматор бросает на `fit` (в т.ч. в одном фолде) → кандидат failed, `run_slice` завершается, leaderboard
без него, WARNING присутствует, прочие модели в leaderboard; **успешные кандидаты имеют одинаковую OOF-маску**
(фикс R1-B1: тест, что `rank` не сравнивает разные покрытия); баг в `align_proba`/`project_for_metric`
(наш код) **пробрасывается** наружу, не превращается в «failed» (фикс R1-3a); **все** кандидаты бросают →
`FitFailedError` со сводкой; `SliceResult.failed` содержит id+причину; успешный прогон без падений →
`failed == []` (back-compat поведения).

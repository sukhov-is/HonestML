# ADR-0098: Конфигурируемое взвешивание периодов в скоре и значимости

- **Статус:** Accepted (реализован, этап 3 — 2026-06-23)
- **Дата:** 2026-06-23
- **Драйверы:** D-3 (корректное взвешивание неравных периодов); FR-6, NFR-2/4.
  Расширяет ADR-0026 (значимость, блочный bootstrap) и ADR-0010 (pooled-OOF лидерборд).

## Контекст
Скор слайса — **pooled-OOF**: один `metric.score` по всем валидным OOF-строкам (`slice.py:818-825`).
Блочный bootstrap значимости ресэмплит **целые фолды** (`block_index`) с **равной вероятностью
блока** (`significance.py:83-88`). Под периодной схемой (ADR-0096) периоды **неравны по числу
строк**: pooled-скор перевешивает крупные месяцы, тогда как bootstrap уже трактует периоды
равновесно — точечный скор и тест значимости рассогласованы. Пользователь выбрал
**конфигурируемость** обеих семантик.

## Рассмотренные варианты
1. **Только pooled (как сейчас).** Минимум кода, но «помесячный» вывод недостоверен на неравных
   периодах; рассогласование со block-bootstrap сохраняется. Недостаточно (выбор пользователя).
2. **Только per-period (заменить pooled).** Меняет смысл существующего лидерборда для всех схем —
   нарушает NFR-5 (обратная совместимость). Отвергнут.
3. **Knob `weighting ∈ {pooled, period}`, дефолт `pooled`.** Аддитивно, текущее поведение —
   дефолт; period-режим согласует точечный скор с равновесным block-bootstrap. **Выбран.**

## Решение

### 1. `CVConfig.weighting: Literal["pooled","period"] = "pooled"` (core, аддитивно)
- Гейт в `build`: `weighting="period"` требует **time-ordered** схему (`timeseries` или
  `timeseries_period`) — нужен `block_index` (фолд/период на строку). Иначе `ConfigError`.

### 2. Точечный скор (application/slice.py)
- Выделить хелпер `_score_weighted(y, pred, mask, block_index, metric, sw, weighting)`:
  - `pooled` → текущий путь (один `metric.score` по `[mask]`), без изменений.
  - `period` → метрика считается **по каждому блоку** `block_index` (= период/фолд) на валидных
    строках блока; **среднее по блокам с конечной метрикой** (макро по периодам). Блок с
    неопределённой метрикой (один класс для roc_auc → `ValueError`) отбрасывается (паттерн NaN-drop
    в `_delta`, `significance.py:104-109`, R-6). Все блоки невалидны → `nan` (как при пустой маске).
- **Проводка (F3):** точечный скор кандидата считается в `_run_candidate` (`slice.py:818-825`), куда
  сейчас НЕ передаются `block_index`/`weighting` — протянуть оба (`block_index` строится один раз в
  `run_slice:354`; `weighting` приходит config-значением). Цепочка: `CVConfig.weighting` →
  `Components` (новое поле) → `facade.run_slice(...)` (новый аргумент, прецедент `selection`/
  `refinement_min_oof`) → `_run_candidate`. Скор кандидата и скор анкера/band считаются одним
  хелпером — согласованность лидерборда и band конструктивна, а не декларативна.
- **Анти-survivorship (F7):** при `weighting='period'` сравниваемые кандидаты скорятся на **общем
  наборе валидных периодов** — блоки, где у любого кандидата метрика не определена, исключаются у
  всех (паттерн `_common_mask`, `selection_policy.py:145-154`, перенесённый на уровень блоков).
  Иначе кандидаты сравнивались бы на разных наборах периодов (нарушение инвариантности пула,
  ADR-0026 §2). Число дропнутых периодов → в манифест (NFR-8).

### 3. Значимость (adapters/significance.py)
- `BootstrapSignificanceTest` получает поле `aggregate: Literal["pooled","period"] = "pooled"`
  (деталь адаптера, **не часть порта** `SignificanceTest` — осознанная асимметрия с `seed`/`n_boot`,
  которые в порте; `aggregate` ставится только на bootstrap-тест, не на `NoSignificanceTest`, F16).
  При `block_index is not None`:
  - `pooled` (текущее) — `_delta` считает метрику на **конкатенации** строк выбранных блоков.
  - `period` — набор валидных (конечных) блоков фиксируется **до** ресэмпла; ресэмпл — по этим
    блокам; для каждого выбранного блока метрика считается **отдельно**, Δ = среднее per-block
    разностей. Тот же фиксированный набор блоков у обоих кандидатов (анти-survivorship, F7); та же
    RNG/seed — воспроизводимость и инвариантность к пулу сохранены (ADR-0026 §2).
- **Гейт числа блоков (F2/G3, BLOCKER):** блок = тест-фолд (`np.unique(block_index)`), т.е. их
  число = `n_splits` (типично 3–6 в помесячном/inner-CV сценарии). Существующий
  `n_boot*alpha < _MIN_TAIL` (`significance.py:52`) проверяет хвост ресэмплов, но НЕ число блоков:
  при 2–3 блоках перцентильный CI вырождается в дискретную решётку → фиктивная уверенность. Гейт с
  **явной константой** `_MIN_BLOCKS` рядом с `_MIN_TAIL`, **одна** реакция по диапазону (тестируемо):
  `n_valid_blocks < 4` → `ConfigError` с понятным текстом (fail-fast, как «мёртвый зноб невозможен»);
  `4 ≤ n_valid_blocks < _MIN_BLOCKS` (ориентир 8) → тест **выполняется** + WARNING «широкий band из-за
  малого числа периодов». Порог 8–10 как безусловный отказ отвергнут — задушил бы `weighting='period'`
  на целевом помесячном горизонте (G3).
- Сборка в `composition`: `aggregate` берётся из `CVConfig.weighting` (только для time-ordered);
  `BootstrapSignificanceTest` конструируется после резолва схемы (`resolved` уже доступен в `build`).

### 4. Наблюдаемость и границы (G7, G12)
- **Отражение в отчёте (G7):** при `weighting='period'` число на лидерборде меняет смысл (макро по
  периодам), поэтому `weighting` и `n_periods_used` (число валидных периодов) сериализуются в
  отчётный CV-блок (тот же канал `SliceResult.cv_split`, ADR-0096 §4) — НЕ в `LeaderboardEntry`
  (`extra='forbid'`) и не в артефакт. Пользователь видит, каким способом и на скольких периодах
  посчитан скор.
- **FS `null_block_*` независим от `period` (G12):** структура блоков null-importance
  (`null_block_mode`/`null_block_window`, ADR-0055) имеет собственный механизм и НЕ наследует
  `period` CV-схемы — это разные оси (CV-нарезка vs перестановочные блоки FS). Проброс `period` в
  FS-null вне области; зафиксировано здесь, чтобы снять dangling-ссылку из ADR-0096 §3.

## Последствия
- **Положительные:** «помесячный» скор честен (каждый период весит одинаково); точечный скор
  согласован с block-bootstrap; pooled остаётся дефолтом — существующие прогоны не меняются (NFR-5).
- **Отрицательные/компромиссы:** макро-по-периодам метрика — иной оценщик, чем pooled (для
  недекомпозируемых метрик это осознанная макро-семантика, документируется, R-6); +поле в тесте
  значимости и ветка в `_delta` (аддитивно).
- **Влияние на слои:** `core` (поле config), `application` (хелпер скора), `adapters`
  (ветка значимости), `composition` (проводка `aggregate`). Слои не нарушены; `core` остаётся
  без тяжёлых ML-зависимостей (метрика инжектится).

## Проверки
- На синтетике с неравными периодами `pooled` и `period` дают разные детерминированные числа;
  `period`-скор равен ручному среднему per-period метрик (FR-6).
- Блок с одним классом исключается из среднего, расчёт не падает (R-6); два кандидата с разным
  набором валидных блоков сравниваются на ОБЩЕМ наборе периодов (F7).
- **Малое число блоков под `aggregate='period'`** → видимая реакция (консервативное «эквивалентны»
  + предупреждение или `ConfigError`), не молчаливый уверенный band (F2).
- `weighting="period"` при не-time-ordered схеме → `ConfigError` (FR-9).
- Дефолт `pooled`: скор и значимость байт-в-байт как до фичи (NFR-5, регресс существующих тестов).

# ADR-0094: Поверхность конфигурации — поле потолка на `Task`

- **Статус:** Accepted (реализовано 2026-06-23)
- **Дата:** 2026-06-23
- **Драйверы:** D-3 (sound defaults + opt-out), D-5 (воспроизводимость/откат), FR-4, NFR-6, R-5

## Контекст
Потолок кардинальности (ADR-0093) должен быть настраиваемым с разумным дефолтом и **opt-out**, в
духе библиотеки (sound defaults, no over-engineering). Силы: гейт живёт и решает в **core**
(ADR-0092), значит конфиг должен быть доступен в core без пересечения слоя; значение должно
round-trip'иться в manifest и входить в run-fingerprint (R-5, FR-4, NFR-6).

## Рассмотренные варианты
1. **Поле на `Task` (`core/task.py`).** `Task` **уже** владеет порогами кардинальности auto-typing
   (`numeric_cat_max_unique`, `numeric_id_rate`, `numeric_id_min_unique`, `string_id_*`). Новый
   потолок — та же ось кардинальности, тот же слой. Плюсы: в-слое (core), рядом со смысловыми
   соседями; `Task` сериализуется в manifest и хешируется в `compute_run_fingerprint`
   (`task.model_dump()`), значит FR-4/NFR-6/R-5 закрываются «бесплатно»; opt-out — просто значение
   поля. Минусы: расширяет `Task` ещё одним полем (приемлемо — однородно с существующими).
2. **Поле на `FEConfig`.** Минусы: FEConfig — про **генерацию** фич (`target_encoding`,
   `intersections`), а гейт — про **роутинг** уже-категориальных; FEConfig не достигает чисто пути
   `refit_best`. Семантически не туда.
3. **Новый конфиг-объект (`NativeCategoricalConfig`).** Минусы: целый объект ради одного статического
   порога — over-engineering, противоречит `<scope_constraints>`.
4. **Хардкод-константа в core (без конфига).** Минусы: нет opt-out (FR-4 требует возможности
   поднять/отключить) — отвергнут.

## Решение
**Вариант 1.** Потолок — **одно поле на `Task`**, имя **`native_cat_max_unique`** (согласовано со
словарём соседей `numeric_cat_max_unique`/`string_id_*`; ревью R1-#8/R3-#3 — имя фиксируем сейчас,
т.к. оно входит в fingerprint/manifest и переименование позже само ломает ключ). Тип —
**`int | None`**, дефолт **`64`** (эмпирически пиннут, ADR-0093); **`None` ⇒ гейт выключен** (все
категориальные нативно — явный, типобезопасный opt-out, без магического «∞»/большого числа). Гейт в core читает
порог из `Task` (доступен там же, без пересечения слоя). Поскольку `Task` сериализуется в manifest
и входит в run-fingerprint:
- **opt-out** — поднять значение (вернуть «всё нативно») или опустить; ничего больше не трогать (FR-4);
- **round-trip** — значение в manifest, грузится с артефактом;
- **воспроизводимость** — смена значения/дефолта меняет `run_fingerprint` ⇒ нет тихого stale-кэша и
  расхождения роутинга между прогонами (NFR-6, R-5).

## Последствия
- **Положительные:** минимальная поверхность (одно поле); fingerprint/manifest/serialization
  закрываются переиспользованием существующего контура `Task`; однородно с auto-typing-порогами.
- **Отрицательные / компромиссы:** `Task` растёт на поле; смена **дефолта** в будущем minor —
  поведенческое изменение, но покрыта fingerprint'ом (другой ключ кэша) и semver-политикой.
- **Влияние на слои/границы:** `Task` — core-сущность; чтение порога гейтом в core не пересекает
  слой; никакой ML.

## Проверки
`test_task.py`/`test_config*`: дефолт **в обоснованной полосе** (нижняя ≥ существующего
`numeric_cat_max_unique`, верхняя — «десятки», ADR-0093) **и** демотит high-card-фикстуру (конкретная,
falsifiable-проверка вместо расплывчатого «разумен» — ревью R2-#вердикт); `test_run_fingerprint`: два
значения порога → разные fingerprint'ы, одно → одинаковый (NFR-6); round-trip `Task` через manifest
(существующий контур). FR-4: `native_cat_max_unique=None` ⇒ роутинг идентичен до-гейтному (NFR-3).

> Совместимость: `Task` — `extra="forbid"` (в отличие от `FeatureSchema`/`CategoryTable` с
> `extra="ignore"`). Старый manifest без поля грузится (отсутствующий ключ → дефолт), но manifest,
> записанный **новой** библиотекой, **не** прочтётся **старой** (`extra="forbid"` отвергнет
> неизвестный ключ) — pre-existing-поведение `Task`, не вводится этой дельтой; отмечено в operational.
